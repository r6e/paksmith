# Generating UE-format fixtures for paksmith

This document covers how to use the **official Unreal Engine
toolchain** (`UnrealPak`, `UnrealEditor` commandlets, `RunUAT
BuildCookRun`) to produce every fixture paksmith needs for testing,
across all 9 phases of `docs/plans/ROADMAP.md`.

The intent is reproducibility: a maintainer should be able to read one
row of the [generation matrix](#generation-matrix), follow the matching
[phase recipe](#phase-recipes), and end up with a byte-stable fixture
that pins a specific wire-format edge case.

> **Scope note.** Paksmith already ships several dozen synthetic pak fixtures
> generated by `crates/paksmith-fixture-gen` (cross-validated against
> [`trumank/repak`](https://github.com/trumank/repak)) and 13 synthetic
> uasset fixtures (cross-validated against `unreal_asset`). This guide
> documents the **complementary** track: how to produce UE-cooked
> ground-truth fixtures using Epic's own tooling, so paksmith's parser
> can be validated against real engine output rather than against
> another parser. Both tracks are needed; neither alone is sufficient.

## Scope

This doc covers fixture generation for:

- All 11 pak format versions (V1 through V11) — see
  [`crates/paksmith-core/src/container/pak/version.rs`][src-pak-version]
  for the in-tree mapping.
- All paksmith-supported UE asset versions —
  `FileVersionUE4 ∈ [504, 522]` and UE5 1001–1010 plus
  `LegacyFileVersion ∈ {-7, -8, -9}`. See
  [`crates/paksmith-core/src/asset/version.rs`][src-asset-version] for
  the authoritative constant list.
- Compression backends: uncompressed, zlib (always available), Oodle
  (UE 4.27+).
- AES encryption: per-pak index encryption + per-entry encryption,
  with the `Crypto.json` key file UE 4.20+ uses.
- IoStore containers (`.utoc` / `.ucas` / `.uptnl`).
- Versioned and unversioned property serialization (the
  `PKG_UnversionedProperties` package flag).
- All asset-shape edge cases: containers (arrays/maps/sets), extended
  types (`FText`, `FSoftObjectPath`), companion files (`.uexp` and
  `.ubulk` bulk-data sibling stitching).

For Phases 3+ (texture/mesh/audio export), this doc points at the
asset types each handler needs as cooked input; the export side is
handler-specific and lives in the export-phase plans.

## Legal and licensing posture

Paksmith follows the same posture as [CUE4Parse][cue4parse] and
[trumank/repak][repak]:

- **No bundled Epic-licensed binary content.** The repo ships
  synthetic fixtures (hand-rolled bytes via
  `crates/paksmith-fixture-gen`) and may ship UE-cooked output **only
  when the cooked input itself is synthetic** — e.g. an empty UE
  project with a hand-authored Blueprint asset, cooked with
  UnrealPak. No Marketplace assets, no engine starter content, no
  game-derived material.
- **No third-party game content, ever.** Paksmith fixtures must
  originate from maintainer-authored synthetic input cooked by the
  maintainer's own UE install. Never extract from, derive from, or
  commit content sourced from a shipped third-party game — that
  material is governed by the publishing game's EULA (not Epic's UE
  EULA) and may also implicate DMCA §1201 anti-circumvention. This
  applies equally to `.pak` payloads, `.uasset` bytes, AES keys, and
  `.usmap` mapping files.
- **EpicGames/UnrealEngine source.** The Unreal Engine repository is
  access-gated (requires Epic EULA acceptance and Epic-account
  linkage). `https://github.com/EpicGames/UnrealEngine` returns HTTP
  404 to unauthenticated clients. This guide therefore cannot quote
  UE source directly; it cites community references (CUE4Parse,
  repak, FModel, UE4SS) that themselves derive from UE source under
  the Epic EULA.
- **Oodle SDK.** Oodle (RAD Game Tools, now Epic-owned) is bundled
  with UE 4.27+ for use by licensees. Redistribution of the Oodle
  shared library outside an Epic-licensed pipeline is restricted;
  paksmith **must not** vendor `oo2core_*.dll` /
  `liboo2corelinux64.so.9` / equivalent. Phase 8's IoStore reader
  should expect users to supply the library at runtime (see
  [#245][issue-245] audit; FModel uses the same posture). [^oodle-licensing]
- **Crypto keys in fixtures.** Test fixtures use **synthetic test
  keys generated for paksmith** (all-zeros, all-ones, or a documented
  hand-picked 32-byte pattern — see [Encryption](#encryption)). Never
  use a key recovered from a shipped third-party game. Real keys live
  in user-supplied profiles (Phase 5) that paksmith reads at runtime;
  the maintainer never handles them.
- **EULA.** Anyone running the tooling commands in this doc is
  responsible for accepting the Unreal Engine EULA at install time.

## UE version / wire-format reference

The two-axis mapping below is the authoritative reference for which
UE engine releases produce which wire-format combinations. Paksmith's
internal source of truth for both axes is:

- Pak format: [`PakVersion`][src-pak-version] (variants `Initial`
  through `Fnv64BugFix`).
- Asset format: [`AssetVersion`][src-asset-version] and the
  `VER_UE4_*` / `VER_UE5_*` constants in the same file.

### Pak version → UE engine version → key change

Pulled from [`PakVersion`][src-pak-version] (definitive). See
[CUE4Parse `EGame`][cue4parse-egame] for the parser-side mirror.

| Wire | `PakVersion` variant | UE version | Key change |
|------|----------------------|------------|------------|
| 1 | `Initial` | UE 4.0 | First pak format |
| 2 | `NoTimestamps` | UE 4.3 | Per-entry timestamps removed |
| 3 | `CompressionEncryption` | UE 4.4 | Compression + AES support |
| 4 | `IndexEncryption` | UE 4.16 | Index encryption |
| 5 | `RelativeChunkOffsets` | UE 4.20 | Compression block offsets become entry-relative |
| 6 | `DeleteRecords` | UE 4.21 | Delete records (patching) |
| 7 | `EncryptionKeyGuid` | UE 4.22 | Encryption-key GUID in footer |
| 8 | `V8A` | UE 4.22 (brief window) | 4-slot FName comp table, **u8** per-entry compression byte |
| 8 | `V8B` | UE 4.23–4.24 | 5-slot FName comp table, **u32** per-entry compression byte |
| 9 | `FrozenIndex` | UE 4.25 | Optional frozen-index format |
| 10 | `PathHashIndex` | UE 4.26 | Path-hash + encoded directory index replaces flat index |
| 11 | `Fnv64BugFix` | UE 4.27 → UE 5.x | FNV-64 hashing bug fix; current format |

V8A and V8B share the on-disk version byte (`8`); see the
[`PakVersion` doc comment][src-pak-version] for how the footer's
slot-count post-corrects the variant. UE 5.0 through UE 5.x continue
emitting V11 pak files — Epic has not bumped the format since 4.27. [^v11-still-current]

### UE engine version → `FileVersionUE4` / `FileVersionUE5`

Pulled from [`asset::version`][src-asset-version] (constants are
**wire-format gates** — a field appears in the stream only when the
file's reported version is ≥ the constant). UE5 introduced a separate
`FileVersionUE5` axis layered on top of the existing UE4 axis.

| UE version | `FileVersionUE4` | `FileVersionUE5` | `LegacyFileVersion` | Notable wire-format gates that fire |
|------------|------------------|------------------|---------------------|-------------------------------------|
| UE 4.21 | 503 | — | -7 | (below paksmith floor of 504) |
| UE 4.22 | 504 | — | -7 | `NAME_HASHES_SERIALIZED` |
| UE 4.23 | 507 | — | -7 | `PRELOAD_DEPENDENCIES_IN_COOKED_EXPORTS` (cooked) |
| UE 4.24 | 508 | — | -7 | `TEMPLATE_INDEX_IN_COOKED_EXPORTS` |
| UE 4.25 | 510 | — | -7 | `ADDED_SEARCHABLE_NAMES`, `64BIT_EXPORTMAP_SERIALSIZES` (511) |
| UE 4.26 | 516 | — | -7 | `ADDED_PACKAGE_SUMMARY_LOCALIZATION_ID` |
| UE 4.27 | 522 | — | -7 | (canonical UE4 final — paksmith's reference cooked fixture) |
| UE 5.0 | 522 | 1004 | -8 | UE5 begins; `LARGE_WORLD_COORDINATES` |
| UE 5.1 | 522 | 1008 | -8 | `ADD_SOFTOBJECTPATH_LIST` |
| UE 5.2 | 522 | 1009 | -8 | `DATA_RESOURCES` |
| UE 5.3 | 522 | 1009 | -8 | (no new gate vs 5.2) |
| UE 5.4 | 522 | 1012 | -9 | `SCRIPT_SERIALIZATION_OFFSET` (1010), `PROPERTY_TAG_EXTENSION_AND_OVERRIDABLE_SERIALIZATION` (1011), `PROPERTY_TAG_COMPLETE_TYPE_NAME` (1012); LegacyFileVersion drops to -9 |
| UE 5.5 | 522 | 1013 | -9 | `ASSETREGISTRY_PACKAGEBUILDDEPENDENCIES` |
| UE 5.6 | 522 | 1017 | -9 | `METADATA_SERIALIZATION_OFFSET` (1014), `VERSE_CELLS` (1015), `PACKAGE_SAVED_HASH` (1016), `OS_SUB_OBJECT_SHADOW_SERIALIZATION` (1017) |
| UE 5.7 | 522 | 1018 | -9 | `IMPORT_TYPE_HIERARCHIES` |
| UE 5.8 | 522 | 1018 | -9 | (no new gate vs 5.7) |

`FileVersionUE4` stays at 522 for every UE5 row — UE5 kept the UE4
axis frozen at its UE 4.27 value and evolves on `FileVersionUE5`
instead.

> Cross-reference these UE5 values against [CUE4Parse `EGame`][cue4parse-egame]
> on a per-release basis before pinning a fixture to a sub-revision —
> some minor UE updates have shipped wire-format changes without a
> visible release-note bump.

**Paksmith's current support window.** Paksmith refuses files where
`FileVersionUE5 ≥ FIRST_UNSUPPORTED_UE5_VERSION` (currently **1014**
— see [`asset::summary`][src-asset-summary]). UE 5.4+ packages
(`FileVersionUE5 ≥ 1012`) are therefore rejected today. The fixture
matrix still lists UE 5.4–5.8 rows because paksmith is forward-looking:
fixtures for those versions need to exist when the floor is raised.
Rows above the floor are marked with **(awaiting paksmith support)**.

## Tools

Four executables produce every UE-cooked fixture paksmith needs.

### UnrealPak

**Binary location** (per UE install):

| Platform | Path |
|----------|------|
| Windows | `Engine\Binaries\Win64\UnrealPak.exe` |
| macOS | `Engine/Binaries/Mac/UnrealPak` |
| Linux | `Engine/Binaries/Linux/UnrealPak` |

**What it does.** Packs cooked-asset files into `.pak` archives.
With the right flags it can also drive IoStore container creation
(though most pipelines go through `UnrealEditor -run=IoStore`
instead — see [IoStore](#iostore-utoc--ucas--uptnl)).

**Invocation shapes:**

```
UnrealPak <pak-path> -Create=<response-file> [options]   # build a .pak
UnrealPak <pak-path> <output-dir> -Extract               # extract a .pak (or .utoc/.ucas — see below)
UnrealPak <pak-path> -Test                               # integrity-check
UnrealPak <pak-path> -List                               # list entries
UnrealPak IoStore -CreateGlobalContainer=<path> ...      # build IoStore container set
```

**Pak-side flags** (sources: UE 4.22
[community gist][gist-unrealpak] and
[forum usage thread][forum-unrealpak], confirmed working on later
UE versions per [Z's Blog][zs-blog-pak]):

| Flag | Meaning |
|------|---------|
| `-Create=<response-file>` | Build a pak from a response file (format below) |
| `-Test` | Validate pak integrity |
| `-List` | Print archive entries |
| `-Extract` | Unpack pak contents; expects two positional args `<file> <outdir>` before the flag |
| `-Repack` | Repack an existing pak (mutate compression, encryption, etc.) |
| `-AuditFiles` | Audit pak contents |
| `-WhatsAtOffset <offset>` | Identify what record sits at a byte offset |
| `-TestEncryption` | Smoke-test the encryption configuration |
| `-compress` | Enable compression on entries |
| `-compressionformats=Zlib`, `-compressionformats=Oodle,Zlib` | Pick compression backend(s); first that succeeds wins per entry |
| `-compresslevel=<level>` | Compression level (Oodle: `Kraken`, `Mermaid`, `Leviathan`, `Selkie`) |
| `-blocksize=<bytes>` | Compression block size (default 64 KiB) |
| `-encrypt` | Per-entry encryption on |
| `-encryptindex` | Encrypt the pak's directory index |
| `-cryptokeys=<path>` | Path to a `Crypto.json` key file |
| `-encryptionini=<path>` | Path to a `DefaultEngine.ini` whose `[/Script/CryptoKeys.CryptoKeysSettings]` section holds the key |
| `-order=<file>` | File-ordering specification (controls layout) |
| `-responsefile=<file>` | (under `-Extract`) emit a response file describing the extraction |
| `-Filter=<glob>` | (under `-Extract`) extract only entries matching the glob |
| `-ExtractToMountPoint` | (under `-Extract`) preserve mount-point relative paths |

**IoStore-side dispatch.** UnrealPak switches into IoStore mode
when **any** of these markers appears on the command line:

| Trigger | Meaning |
|---------|---------|
| Positional `IoStore` as first non-option argument | Build IoStore containers from a response file (`-Output=…`, `-ContainerName=…`, `-ResponseFile=…`) |
| `-CreateGlobalContainer=<path>` | Build the global `global.utoc` / `global.ucas` pair at `<path>` |
| `-CreateDLCContainer=<path>` | Build a DLC container |
| `-ListContainer=<path.utoc>` | List entries inside an IoStore container |
| `-ListContainerBulkData=<path.utoc>` | List the container's bulk-data segment |
| `-DiffContainer=<path.utoc>` | Diff two containers |

Community references to a `-CreateIoStoreContainer=` flag are
inaccurate — that spelling does not exist. The actual entry points
are the five flags above plus the positional `IoStore` mode.

**Extraction of `.utoc` / `.ucas`.** Same `-Extract` syntax as a pak.
UnrealPak's extraction dispatcher detects an IoStore container and
routes through the IoStore extraction path automatically — so
`UnrealPak MyGame.utoc /tmp/out -Extract -cryptokeys=Crypto.json`
works end-to-end. Generating an output response file from an
IoStore container is **not supported** (the dispatcher logs a
warning if you pass `-responsefile=`).

**Response file format** (one entry per line, source:
[Z's Blog][zs-blog-pak], [forum thread][forum-unrealpak]):

```
"C:/abs/path/to/file.uasset" "../../../MyProject/Content/file.uasset" -compress
"C:/abs/path/to/file.uexp"   "../../../MyProject/Content/file.uexp"   -compress
"C:/abs/path/to/file.ubulk"  "../../../MyProject/Content/file.ubulk"
```

Each line has three fields: absolute source path, relative virtual
path (the path the pak's mount-point sees), optional per-entry flags
like `-compress`.

### UnrealEditor commandlets

The editor binary doubles as a script runner via `-run=<commandlet>`.

**Binary location:**

| Platform | Path |
|----------|------|
| Windows | `Engine\Binaries\Win64\UnrealEditor-Cmd.exe` (UE5) or `UE4Editor-Cmd.exe` (UE4) |
| macOS | `Engine/Binaries/Mac/UnrealEditor` (UE5) or `UE4Editor` (UE4) |
| Linux | `Engine/Binaries/Linux/UnrealEditor` (UE5) or `UE4Editor` (UE4) |

**Cook commandlet:**

```
UnrealEditor-Cmd <project.uproject> -run=Cook \
    -targetplatform=<Plat1>[+<Plat2>] \
    [-map=<Map1>[+<Map2>]] \
    [-CookCultures=en+fr] \
    [-iterate] \
    [-cookonthefly] \
    [-multiprocess] \
    [-NODEV] \
    [-skipeditorcontent]
```

The `-targetplatform` values that matter for paksmith fixtures are
`WindowsNoEditor` / `Windows` (UE4 vs UE5 spelling),
`LinuxNoEditor` / `Linux`, and `MacNoEditor` / `Mac`. Cooked output
lands in `<Project>/Saved/Cooked/<Platform>/<Project>/Content/...`.

Sources: [ikrima Gamedev Guide][ikrima-cook], [old UE4 wiki cooking
on Linux][nerivec-linux], [community forum cook-commandlet
thread][forum-cook]. Flag semantics:

| Flag | Meaning |
|------|---------|
| `-targetplatform=<P>` | Required. Platform to cook for. |
| `-map=<M>` | Cook only specified maps (default: cook all) |
| `-CookCultures=<langs>` | Cook only listed localization cultures |
| `-iterate` | Reuse prior cooked output where unchanged |
| `-cookonthefly` | Stream content (client connects to running editor) |
| `-multiprocess` | Distribute cook across worker processes (UE 5.2+) |
| `-NODEV` | Exclude `/Game/Developers/...` |
| `-skipeditorcontent` | Drop `/Engine/Editor/...` |

**Other useful commandlets:**

- `-run=DumpAssetRegistry` — dumps the asset registry to a file
- `-run=ResavePackages` — re-saves uasset(s) to refresh format
- `-run=IoStore` — drives IoStore container creation (see
  [IoStore section](#iostore-utoc--ucas--uptnl))
- `-run=DerivedDataCache` — populates the DDC

### RunUAT BuildCookRun (Unreal Automation Tool)

The high-level "do everything" script. Wraps cook + stage + pak +
deploy.

**Wrapper script location:**

| Platform | Path |
|----------|------|
| Windows | `Engine\Build\BatchFiles\RunUAT.bat` |
| macOS | `Engine/Build/BatchFiles/RunUAT.sh` |
| Linux | `Engine/Build/BatchFiles/RunUAT.sh` |

**Invocation:**

```
RunUAT BuildCookRun -project=<abs-path-to-uproject> \
    -platform=<Win64|Mac|Linux> \
    -clientconfig=Development \
    -cook -build -stage -pak \
    [-iostore] \
    [-compressed] \
    [-encryptini] \
    [-archive -archivedirectory=<dir>] \
    [-noP4]
```

**Flag semantics** (sources: [ikrima UAT reference][ikrima-uat],
[old UE4 wiki packaging guide][nerivec-package],
[ikrima cook guide][ikrima-cook]):

| Flag | Meaning |
|------|---------|
| `-project=<.uproject>` | Required. Absolute path to the project file. |
| `-platform=<P>` | Build/cook target platform (`Win64`, `Mac`, `Linux`). |
| `-clientconfig=<Cfg>` | `Debug`, `DebugGame`, `Development`, `Test`, `Shipping`. |
| `-cook` | Run the cook step. |
| `-skipcook` | Use existing cooked data without rebuilding. |
| `-build` | Compile the project's code module. |
| `-stage` | Copy cooked output into a staging dir. |
| `-pak` | Package staged content into a `.pak`. |
| `-iostore` | Also produce IoStore containers (UE 4.27+; requires `-pak`). |
| `-skippak` | Use existing pak file. |
| `-compressed` | Enable pak compression (uses `PakFileCompressionFormats` from project settings). |
| `-encryptini` | Encrypt `.ini` files inside the pak. |
| `-signpak=<keyfile>` | Sign pak entries with the given key. |
| `-cookflavor=<f>` | Texture/etc. flavor (e.g. `ASTC` for Android). |
| `-archive` | Copy the staged build to `-archivedirectory`. |
| `-archivedirectory=<dir>` | Destination for `-archive`. |
| `-allmaps`, `-maps=A+B` | Map selection. |
| `-noP4` | Bypass Perforce integration. |
| `-unattended` | Non-interactive mode (suppress prompts). |

**Output layout:**

```
<Project>/Saved/Cooked/<Platform>/        # Raw cook output (intermediate)
<Project>/Saved/StagedBuilds/<Platform>/  # Staged build (post-pak)
    <Project>/Content/Paks/
        <Project>-<Platform>.pak
        # IoStore (if -iostore):
        <Project>-<Platform>.utoc
        <Project>-<Platform>.ucas
        global.utoc                 # Global container (per build)
        global.ucas
```

### UnrealBuildTool (UBT)

Compiles C++ project code into the engine binary used for cooking.
Invoked indirectly via `RunUAT … -build` and rarely needed in
isolation for fixture generation **unless** the project has a C++
module. Blueprint-only projects skip UBT entirely.

| Platform | Path |
|----------|------|
| Windows | `Engine\Binaries\DotNET\UnrealBuildTool\UnrealBuildTool.exe` |
| macOS / Linux | `Engine/Build/BatchFiles/<Mac\|Linux>/Build.sh` (wrapper) |

For Blueprint-only fixture projects (recommended for reproducibility
— no C++ toolchain dependency), skip UBT entirely.

## Per-platform setup

| Topic | Windows | macOS | Linux |
|-------|---------|-------|-------|
| Editor binary | First-class via Epic Launcher / GitHub | First-class | Tier-2 — Epic-supported but source-only install for some UE versions |
| Cook commandlet | Yes | Yes | Yes |
| UnrealPak | Yes | Yes | Yes |
| RunUAT | `.bat` | `.sh` | `.sh` |
| Cook targets you can produce | `Windows`, `Linux`, `Android`, `iOS`*, `Mac`* | All Windows targets + native Mac/iOS | `Linux`, `Windows`*, `Android` |
| Oodle DLL | `oo2core_9_win64.dll` shipped with UE 4.27+ | `liboo2coremac64.dylib` | `liboo2corelinux64.so.9` |

`*` Cross-compiling to platforms whose SDK isn't installed will fail
during the cook step. Build paksmith fixtures on the host that
matches your target platform when feasible.

**macOS caveat.** UE on Apple Silicon requires UE 5.2+ for native
arm64; earlier versions require Rosetta 2. Cooked output is
architecture-independent (the cook itself runs on the host, target
platform is selected via `-targetplatform`), so this affects only
your local UE install — not the fixture bytes.

**Linux caveat.** Some UE source releases ship without prebuilt Linux
binaries — you must run `Setup.sh` and `GenerateProjectFiles.sh` and
then build the editor yourself. See the [old UE4 wiki Linux cook
guide][nerivec-linux] for the prerequisite check (Engine/Binaries/Linux
should contain `UnrealEditor`, `UnrealFrontend`, `ShaderCompileWorker`,
`UnrealPak`).

## Minimal cooking project template

The reproducibility goal is: any maintainer with a local UE install
should be able to regenerate any fixture row in the matrix from a
**known-good minimal project** plus a documented set of CLI flags.
The minimal project lives outside the paksmith repo (it's not
license-clean to commit cooked output of even a synthetic project to
a public Rust crate without scrutiny — see [Legal
posture](#legal-and-licensing-posture)). The maintainer creates the
project once on their workstation.

### Minimal `.uproject` (Blueprint-only)

```json
{
  "FileVersion": 3,
  "EngineAssociation": "5.4",
  "Category": "",
  "Description": "Paksmith fixture cook project — Blueprint-only, no C++ module."
}
```

Adjust `EngineAssociation` to match the UE version you're cooking
with. Place this file as `PaksmithFixtures/PaksmithFixtures.uproject`.

### Minimal `Config/DefaultEngine.ini`

```ini
[/Script/Engine.RendererSettings]
r.SkinCache.CompileShaders=True

[/Script/UnrealEd.ProjectPackagingSettings]
bUseIoStore=False         ; set True to emit .utoc/.ucas instead of .pak
bCompressed=False         ; True to enable -compressed during pak cook
PakFileCompressionFormats=  ; leave empty for none; set "Zlib" or "Oodle,Zlib"
```

### Minimal `Config/DefaultGame.ini`

```ini
[/Script/EngineSettings.GeneralProjectSettings]
ProjectName=PaksmithFixtures
ProjectID=00000000000000000000000000000000
```

### Crypto config (for encryption fixtures)

When generating an encrypted-pak fixture, drop a `Crypto.json` into
the project root or pass it explicitly via `-cryptokeys=`. See
[Encryption](#encryption) for the file schema.

### Recommended content

For Phase 2 fixtures, author Blueprint assets directly in the editor:

- `BP_BoolField.uasset` — Blueprint with one `bool` variable (Phase 2b)
- `BP_Containers.uasset` — Blueprint with `TArray<int32>`, `TMap<FString, int32>`, `TSet<FName>` (Phase 2c)
- `BP_ExtendedTypes.uasset` — Blueprint referencing an `FText`
  literal and an `FSoftObjectPath` (Phase 2d)
- `T_Test.uasset` — A Texture2D (Phase 2e, Phase 3 texture export)
- `SM_Cube.uasset` — A Static Mesh (Phase 3 mesh export, Phase 9 viewport)
- `SC_Beep.uasset` — A SoundCue / SoundWave (Phase 3 audio export)

Each authored asset, when cooked, generates a `.uasset` + `.uexp`
pair (plus `.ubulk` for assets with bulk data like textures).

## Generation matrix

The centerpiece. Each row pins one specific wire-format combination
paksmith's test suite needs. **`Source` shows whether the fixture
already exists as a synthetic in-repo artifact, or needs to be
freshly cooked.** **`Verify` flags any row where the recipe contains
a claim that should be cross-checked against `UnrealPak -help` or a
known-good reference parser.**

| # | Phase | Fixture | UE ver | Tool | Command summary | Output | Source | Verify? |
|--:|-------|---------|--------|------|-----------------|--------|--------|---------|
| 1 | 1 | `real_v3_minimal.pak` | 4.4–4.15 | UnrealPak | `-Create=resp -compress=none` (V3) | `*.pak` | already in tree (repak) | no |
| 2 | 1 | `real_v6_minimal.pak` | 4.21 | UnrealPak | `-Create=resp` (V6) | `*.pak` | already in tree (repak) | no |
| 3 | 1 | `real_v7_minimal.pak` | 4.22 | UnrealPak | `-Create=resp` (V7) | `*.pak` | already in tree (repak) | no |

## Phase recipes

### Phase 1 — Pak format coverage (V3–V11)

Rows 1–12 in the matrix. Matrix rows 1–8 correspond to the
`real_v{3,6,7,8a,8b,9,10,11}_minimal.pak` fixtures — already in tree under
`tests/fixtures/`, generated by `crates/paksmith-fixture-gen` against

**lack real cooked-UE provenance**.

The four `cooked_v11_*` rows (9–12) are the UE-cooked complement:
they exercise V11 (the only pak version any current UE ships) with
the four compression × encryption permutations.

**Recipe for row 9 (cooked V11, zlib):**

1. Create the minimal project (see [Minimal cooking project
   template](#minimal-cooking-project-template)).
2. Set `Config/DefaultGame.ini`:
   ```ini
   [/Script/UnrealEd.ProjectPackagingSettings]

   bUseIoStore=False
   ```

3. Run:
   ```sh
   RunUAT.sh BuildCookRun \
       -project=/abs/PaksmithFixtures/PaksmithFixtures.uproject \

       -clientconfig=Development \

       -archive -archivedirectory=/tmp/out \
       -noP4 -unattended
   ```

4. Pak lands at `Saved/StagedBuilds/Linux/PaksmithFixtures/Content/Paks/PaksmithFixtures-Linux.pak`.
5. Rename + copy into `tests/fixtures/cooked_v11_zlib.pak`.

**Recipe for row 10 (cooked V11, Oodle):** swap step 2's

**Recipes for rows 11–12 (encryption):** see

### Phase 2a — UAsset structural header

Rows 13–19. Each row pins one wire-format gate in
[`asset::version`][src-asset-version]. The synthetic fixtures are
generated by `crates/paksmith-fixture-gen` and pass field-level
cross-validation against `unreal_asset` (where the parser's API
permits — see [`crates/paksmith-core/src/asset/version.rs`][src-asset-version]

To produce the UE-cooked complement of row 17 (UE 4.27 canonical):

1. Install UE 4.27.
2. Create the minimal project, set `EngineAssociation` to `4.27`.
3. Author one Blueprint asset (any class — `Actor` subclass with a
   string variable is sufficient).
4. Run:
   ```sh

       -run=Cook -targetplatform=LinuxNoEditor -unattended -nullrhi
   ```

5. The `.uasset` + `.uexp` land in
   `Saved/Cooked/LinuxNoEditor/PaksmithFixtures/Content/...`.
6. Copy the pair into `tests/fixtures/` and verify against paksmith.

## Compression backends

### Zlib (built-in, always available)

Default for every UE pak fixture that needs compression. Project
setting:

```ini
[/Script/UnrealEd.ProjectPackagingSettings]
bCompressed=True
PakFileCompressionFormats=Zlib
```

CLI for UnrealPak:

```
UnrealPak my.pak -Create=resp.txt -compress -compressionformats=Zlib
```

### Oodle (UE 4.27+)

Oodle was integrated into UE in **April 2021** as a free-for-UE-licensees
backend (sources: [Epic blog announcement][epic-oodle-blog], [ikrima
oodle integration notes][cbloom-oodle], [Z's Blog UE5 oodle
note][zs-blog-ue5]). It ships with UE 4.27 and all UE5 versions.

**Project settings** (sources: [Z's Blog][zs-blog-ue5], [reduce
package sizes community tutorial][epic-oodle-tutorial]):

```ini
[/Script/UnrealEd.ProjectPackagingSettings]
bCompressed=True
PakFileCompressionFormats=Oodle
PakFileAdditionalCompressionOptions=-compressionblocksize=256KB
PakFileCompressionMethod=Kraken
PakFileCompressionLevel_Distribution=7
PakFileCompressionLevel_TestShipping=5
PakFileCompressionLevel_DebugDevelopment=3
```

`PakFileCompressionFormats` accepts a comma-separated list — entries
that fail to compress (e.g. already-compressed data) fall through to
the next format. A common setting is `Oodle,Zlib` so already-zipped
chunks still compress with zlib if Oodle declines.

**Compression methods** (Oodle):

- `Kraken` — balanced, good general-purpose default
- `Mermaid` — faster decompression, slightly worse ratio
- `Selkie` — fastest decompression, weakest ratio
- `Leviathan` — strongest ratio, slowest

**Per-platform Oodle DLL** ships in
`Engine/Plugins/Compression/OodleData/Sdks/.../<platform>/` and is
loaded at runtime by the engine and by UnrealPak.

**Recompile caveat.** Per [Z's Blog][zs-blog-ue5]:

> "Enabling Oodle during the default packaging requires adding the
> OodleData plugin dependency to UnrealPak, which necessitates
> recompiling UnrealPak."

If the UE distribution you installed didn't bundle Oodle into
UnrealPak (some pre-4.27 binary distributions), you'll need to
recompile UnrealPak from source.

### gzip, brotli, zstd, lz4

- **gzip** — supported by UE's compression abstraction
  (`FCompression`) but not exposed as a `PakFileCompressionFormats`
  option in stock UE. Not used by any shipping game in practice.
- **lz4** — used inside the **IoStore** path for per-chunk
  compression. Not a pak-format option.
- **brotli** — not in stock UE. Community plugin exists.
- **zstd** — not in stock UE. Community plugin exists (see [Z's Blog
  zstd integration][zs-blog-zstd]).

Paksmith only needs to decompress what UE itself can produce, so
zlib + Oodle covers ~100% of real-world fixtures.

## Encryption

UE supports two layered encryption modes:

1. **Per-entry encryption** — each pak entry's body is AES-encrypted
   with the project's key.
2. **Index encryption** — the pak's directory index is AES-encrypted;
   the entries' bodies may or may not be.

Both modes use **AES-256** with a 32-byte key. Pak version 3
introduced AES support; pak version 4 added index encryption; pak
version 7 added the per-archive encryption-key GUID
([`PakVersion::EncryptionKeyGuid`][src-pak-version]).

### Key file (`Crypto.json`)

Generated by **Project Settings → Crypto → Generate New Encryption
Key** in the editor (UE 4.20+). Stored at the project root or
specified via `-cryptokeys=<path>` to UnrealPak.

Schema (source: [UnrealPakTool sample][unrealpak-tool-crypto]):

```json
{
  "$type": "CryptoSettings, UnrealBuildTool",
  "EncryptionKey": {
    "$type": "EncryptionKey, UnrealBuildTool",
    "Name": "Default",
    "Guid": "00000000000000000000000000000000",
    "Key": "<Base64-encoded 32-byte AES key>"
  },
  "SigningKey": null,
  "bEnablePakSigning": false,
  "bEnablePakIndexEncryption": true,
  "bEnablePakIniEncryption": true,
  "bEnablePakUAssetEncryption": false,
  "bEnablePakFullAssetEncryption": false,
  "bDataCryptoRequired": true,
  "SecondaryEncryptionKeys": []
}
```

**Field semantics:**

- `EncryptionKey.Key` — Base64 of the raw 32-byte key. **Not hex.**
  Decoding the Base64 must yield exactly 32 bytes or UE rejects it.
- `EncryptionKey.Guid` — Per-key GUID; written into the pak v7+
  footer's `encryption_guid` field so consumers can pick the right
  key from multiple loaded keys.
- `bEnablePakIndexEncryption` → drives `-encryptindex` at pak time.
- `bEnablePakUAssetEncryption` → drives `-encrypt` (per-entry).
- `bEnablePakFullAssetEncryption` → encrypts everything (rare).
- `SecondaryEncryptionKeys[]` — additional keys for multi-key paks
  (DLC scenarios).

### Recommended test keys for paksmith fixtures

For deterministic fixture generation, use one of these **publicly
documented** keys (never a real shipped key):

| Key name | Hex | Base64 |
|----------|-----|--------|
| All-zeros | `00 × 32` | `AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA=` |
| All-ones | `FF × 32` | `/////////////////////////////////////////w==` |
| Documented demo | `0123456789ABCDEF0123456789ABCDEF0123456789ABCDEF0123456789ABCDEF` | `ASNFZ4mrze8BI0VniavN7wEjRWeJq83vASNFZ4mrze8=` |

Document the exact key used in each fixture's filename or a
sidecar `.key.txt` so future maintainers can decrypt it.

### Per-entry encryption recipe (row 12)

1. Generate `Crypto.json` per the schema above with
   `bEnablePakUAssetEncryption: true`.
2. Cook assets normally (no encryption flags at cook time).
3. Pak them:
   ```sh
   UnrealPak my.pak -Create=resp.txt -encrypt -cryptokeys=Crypto.json
   ```

### Index-encryption recipe (row 11)

Same as above but use `-encryptindex` instead of `-encrypt`.

Sources for this section: [allcoolthingsatoneplace
UnrealPakTool][unrealpak-tool-crypto] (`Crypto.json` schema reference),
[community forum on encrypted paks][forum-encrypted-pak], [Z's Blog
hot-update][zs-blog-hotupdate].

## IoStore (.utoc / .ucas / .uptnl)

**Introduced.** UE 4.25 Experimental. UE 4.26 Beta. UE 4.27 GA.
UE 5.x ships IoStore by default for cooked-game builds.

**On-disk layout.** Paired files per cooked package set:

- `<Name>.utoc` — **Table of Contents.** Chunk IDs, offsets, sizes,
  compression metadata, encryption flags. Read by the engine first;
  drives all subsequent reads.
- `<Name>.ucas` — **Content-Addressable Storage.** Holds the actual
  chunk bytes referenced by the TOC. 16-byte aligned.
- `<Name>.uptnl` — **Optional package payload** (introduced UE 4.27).
  Holds optional / on-demand bulk data separated from the main `.ucas`.

A cooked build also produces a **global** container pair:

- `global.utoc` / `global.ucas` — engine-internal metadata (script
  objects, name table, etc.) shared across all per-package containers.

(Sources: [EpicGamesJapan UE 4.25 IoStore deck][slideshare-iostore],
[Buckminsterfullerene IoStore method gist][buck-iostore-method],
[retoc README][retoc].)

### Producing IoStore containers

**Path 1 — RunUAT (recommended for fixtures):**

```sh
RunUAT.sh BuildCookRun \
    -project=/abs/PaksmithFixtures/PaksmithFixtures.uproject \
    -platform=Linux \
    -clientconfig=Development \
    -cook -build -stage -pak -iostore -compressed \
    -archive -archivedirectory=/tmp/out \
    -noP4 -unattended
```

The `-iostore` flag requires `-pak` to also be present; both
`bUseIoStore=True` and `UsePakFile=True` (or
`bUsePakFile=True` per UE version) must be set in
`Config/DefaultGame.ini`:

```ini
[/Script/UnrealEd.ProjectPackagingSettings]
UsePakFile=True
bUseIoStore=True
```

(Source: [indxzero UE cvar wiki: `bUseIoStore`][indxzero-buseiostore],
[forum thread on Use Io Store setting][forum-iostore-undocumented].)

Output:

```
<Project>/Saved/StagedBuilds/<Platform>/<Project>/Content/Paks/
    <Project>-<Platform>.pak       # residual pak (ini files etc.)
    <Project>-<Platform>.utoc
    <Project>-<Platform>.ucas
    <Project>-<Platform>.uptnl    # if optional payload was cooked
    global.utoc
    global.ucas
```

**Path 2 — UnrealPak IoStore mode:**

```sh
UnrealPak IoStore \
    -CreateGlobalContainer=<output-path>/global \
    -CookedDirectory=<path-to-Saved/Cooked/Linux> \
    -Commands=<IoStoreCommands.txt> \
    -CookerOrder=<file-order.log> \
    -TargetPlatform=LinuxNoEditor
```

The `IoStore` positional triggers the IoStore container-creation
path. `-CreateGlobalContainer=` builds the `global.utoc` /
`global.ucas` pair; `-CreateDLCContainer=` is the DLC analogue.

`IoStoreCommands.txt` lists per-container build specs:

```
-Output=PaksmithFixtures-Linux -ContainerName=PaksmithFixtures -ResponseFile=PakListIoStore.txt
```

And `PakListIoStore.txt` is a standard response file. (See
[Buckminsterfullerene IoStore gist][buck-iostore-method] for a
worked example.)

The `UnrealEditor-Cmd -run=IoStore` commandlet exists as well and
shares the same flag surface, but UnrealPak's `IoStore` mode runs
without spinning up the full editor and is the lighter-weight path
for fixture generation.

### Extracting IoStore containers

`UnrealPak`'s `-Extract` flow handles `.utoc` / `.ucas` directly —
the extraction dispatcher detects an IoStore container and falls
through to the IoStore extraction path:

```sh
UnrealPak MyGame.utoc /tmp/out -Extract -cryptokeys=Crypto.json
```

Two positional arguments are required: the container path and the
output directory. Encryption is auto-detected from the TOC; supply
`-cryptokeys=` if the container is encrypted. Generating a
`-responsefile=` from an IoStore extraction is not supported (UE
logs a warning and continues).

The alternative `UnrealEditor-Cmd … -run=IoStore` extraction path
exists but goes through the full editor module load — heavier and
unnecessary unless you're scripting against editor-only types.

### Compression in IoStore

Per-chunk compression. Default backends: **Zlib**, **LZ4**, **Oodle**
(when available). Encryption is layered on top of compression
(compress first, then encrypt).

## Unversioned properties (UE5)

**The single most consequential change between UE4 and UE5 cooked
assets.** Affects every Phase 2b–2f parser path.

### What it is

UE5 introduced **unversioned property serialization** as the default
for cooked output. Under unversioned serialization:

- Each property in an exported object is written **without** its
  `FPropertyTag` header (no `Name`, `Type`, `Size`, `ArrayIndex`).
- Reading requires an out-of-band **schema** — the `.usmap` mapping
  file — that says "object class X has fields A, B, C in this order
  with these types."
- The package flag `PKG_UnversionedProperties` (in
  `FPackageFileSummary.package_flags`) marks each package as such.

Without a `.usmap`, **no parser** can read the property bodies.
CUE4Parse, FModel, paksmith, `unreal_asset` all need it.

### When it's on / off

- **UE4** (≤ 4.27): always **versioned** (tagged properties); the
  `PKG_UnversionedProperties` flag is never set.
- **UE5**: cooked output uses **unversioned** by default for every
  shipped-target platform (Windows, Linux, Mac, Android, iOS,
  console targets). Editor-mode (uncooked) packages remain versioned.

**There is no `bUseUnversionedPropertySerialization` setting on
`UProjectPackagingSettings`.** The `PKG_UnversionedProperties` flag
is decided per save by the cooker, driven by the target platform's
own preferences (cooked-platform targets opt in; the editor target
opts out) — not by any project-level config.

This means there is no first-class "force versioned cooks" toggle.
To produce a versioned UE5 cooked package for fixture purposes, the
options are, in order of invasiveness:

1. **Cook as `Editor` target.** Editor builds run with full
   `WITH_EDITORONLY_DATA` and serialize versioned tagged properties.
   This yields uncooked-style output, not a true cooked package —
   useful for comparing versioned vs unversioned wire format on the
   same source asset.
2. **Override `ITargetPlatform::HasEditorOnlyData()`** in a custom
   platform module to return `true` while leaving cook flags
   otherwise unchanged. Requires engine modification.
3. **Hand-flip the `PKG_UnversionedProperties` bit** in a cooked
   `.uasset`'s `FPackageFileSummary::package_flags` post-cook — but
   the property data is already in unversioned (header-stripped)
   layout, so the file becomes unparsable. **Don't do this.**

For Phase 2f fixtures, paksmith generally pairs an unversioned cooked
package with a `.usmap` mapping file (next section). The "versioned
UE5" case is mostly useful for sanity-checking the parser fast path,
not for production usage.

### `.usmap` mapping files

`.usmap` is a separate binary file generated **outside** the UE
build pipeline by a "mappings dumper" that introspects a running
UE-built executable's loaded type information.

**For paksmith fixtures, the dumper runs against the maintainer's
own packaged fixture project — never against a third-party shipped
game.** Package the fixture project to a standalone build (RunUAT
BuildCookRun without `-skipstage`), launch that build with the
dumper attached, and capture `Mappings.usmap`.

Dumper options:

- [**UE4SS dumper**][ue4ss] — runs in-process via DLL injection
  against a packaged UE 4/5 build; default keybind `Ctrl+Numpad 6`
  dumps `Mappings.usmap` next to the target executable.
- [**Dumper-7 / UnrealMappingsDumper**][unreal-mappings-dumper] —
  external dumper, targets UE5.
- [**jmap**][jmap] — newer alternative from trumank.
- Paksmith's eventual Phase 2f implementation will need a `.usmap`
  parser. Format details: see
  [the unofficial modding guide][unofficial-mappings] and
  [The Cutting Room Floor wiki][tcrf-mappings].

`.usmap` files are **not produced by the standard UE cook pipeline**;
they require a packaged build plus a separate dumper pass. For
paksmith fixtures, treat the `.usmap` as a separate artifact paired
with the asset and document both as outputs of the maintainer's own
project.

### Fixture recipes

**Row 28 — Unversioned cooked Blueprint (UE 5.4):**

1. Create a UE 5.4 project with the minimal template.
2. **Verify** the project setting "Use Unversioned Property
   Serialization" is **on** (this should be the default for UE5).
3. Author a Blueprint with one bool variable.
4. Cook: `UnrealEditor-Cmd … -run=Cook -targetplatform=Linux`.
5. The cooked package will have `PKG_UnversionedProperties` set.
6. Pair the fixture with a `.usmap` dumped from the same project
   (using one of the dumpers listed above) so paksmith can read it.

**Row 29 — Versioned cooked Blueprint (UE 5.4):**

Per the [When it's on / off](#when-its-on--off) discussion, there is
no project-setting toggle. Two practical approaches for paksmith:

- **Synthetic Phase 2b fixture** (recommended). The
  `crates/paksmith-fixture-gen` synthetic uasset path emits
  versioned tagged properties unconditionally — use that for the
  versioned-path parser test. It's byte-stable and cross-validated.
- **UE 4.27 cooked Blueprint.** UE4 cooks always emit versioned
  tagged properties (the `PKG_UnversionedProperties` flag did not
  exist before UE5). A UE 4.27 cooked Blueprint exercises the same
  parser fast path as a hypothetical "versioned UE5" package would.

This pair (unversioned UE5 + versioned UE4 or synthetic) is enough
to test Phase 2f's reader: it routes on the `PKG_UnversionedProperties`
bit and falls through to the tagged-property reader otherwise.

## Verification and cross-validation

Once a fixture is generated, verify it parses correctly against
**at least two independent parsers**:

1. **Paksmith** (the parser under test). For pak fixtures:
   `cargo run -p paksmith-cli -- list <fixture.pak>`. For asset
   fixtures: integration tests in
   `crates/paksmith-core-tests/tests/`.
2. **One of:**
   - [CUE4Parse][cue4parse] (C#) — most complete community parser.
   - [trumank/repak][repak] (Rust) — pak-only, fast.
   - [trumank/retoc][retoc] (Rust) — IoStore-only.
   - [`unreal_asset`][unreal-asset] (Rust) — uasset-only, currently
     paired with paksmith for the synthetic fixture matrix.
   - [UEViewer][ueviewer] (C++) — long-standing reference; broad UE
     version coverage.

For pak fixtures, paksmith already has
`crates/paksmith-fixture-gen/src/main.rs` cross-validating against
repak; extend it for new rows. For asset fixtures, use the existing
`cross_validate_with_unreal_asset` harness in
`paksmith-fixture-gen`.

When a fixture's bytes are UE-cooked (not paksmith-synthetic), the
cross-validation goes the **other** direction: paksmith parses UE's
output, and divergence is a paksmith bug (or a wire-format gate
paksmith's parser doesn't yet handle).

## Reproducibility

For every UE-cooked fixture committed to the repo, document:

- **UE version** — exact (e.g. `5.4.4`).
- **Source files** — the `.uasset`(s) that went into the cook
  (synthetic; license-clean).
- **Cook command** — full CLI invocation.
- **Project settings diff** — the relevant lines from
  `DefaultEngine.ini` / `DefaultGame.ini`.
- **Crypto.json** (if encrypted) — full content.
- **`.usmap`** (if unversioned) — same provenance.

Suggested location for fixture metadata:
`tests/fixtures/<name>.recipe.md` or a single
`tests/fixtures/COOKED_FIXTURES.md` index. The exact format can
follow paksmith's existing `tests/fixtures/generate.rs` doc
comments.

Pin to a **specific UE point release** (e.g. `4.27.2`, `5.4.4`).
Minor UE updates have, historically, changed asset wire format
without bumping `FileVersionUE5`.

## Troubleshooting

**`UnrealPak` exits with "Ran out of memory allocating N bytes" on
extraction.** Increase virtual memory / swap on the extraction
drive to roughly double the input file size. (Source: [forum
extraction thread][forum-iostore-extract].)

**Cook commandlet hangs or freezes.** Use `-unattended -nullrhi` to
disable interactive prompts and skip GPU init. The cook still runs
to completion without a graphics device.

**"Failure" notification despite successful cook.** Known UE
behavior — always check the actual cook log
(`Saved/Logs/UnrealEditor.log`) for the canonical status. (Source:
[old UE4 wiki Linux cook][nerivec-linux].)

**Oodle not found at pak time.** Check that
`Engine/Plugins/Compression/OodleData/` exists and contains the
target-platform shared library. Some UE source distributions
predate the Oodle bundling and require a manual integration step.

**`.usmap` mismatch errors at parse time.** The `.usmap` was
generated against a different build than the cooked asset.
Regenerate with the dumper running against the same game version.

**Encrypted pak: "decryption failed" / garbage bytes after the
footer.** Wrong key. Verify the Base64 in `Crypto.json` decodes to
exactly 32 bytes; verify the key GUID in the pak footer matches
the `EncryptionKey.Guid` field.

**Pak version mismatch (e.g. reader claims V11, file is V10).**
Don't trust the `version` field alone for V8 — the footer's
compression-slot count distinguishes V8A from V8B. See
[`PakVersion` doc comment][src-pak-version].
